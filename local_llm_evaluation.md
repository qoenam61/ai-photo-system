## 1. 평가 목적

1. **M4 Mac mini 16GB 환경에서 활용 가능한 최대 성능의 로컬 LLM 선정**
2. 선정 모델로 사진 시스템(v3.2)의 LLM 의존 부분(Layer 2, Layer 4) **완전 대체** 검토
3. **Vision API 비용 $0/월 달성** ⭐ (현재 계획: Haiku 4.5 월 ~$0.6 → 폐기)
4. 트레이딩 시스템과의 메모리 충돌 없는 안정 운영 검증
5. 로컬 LLM을 사진 시스템 외 영역(앨범명·리포트·자연어 질의·캡션 등)에 **확장 활용**하여 AI 가치 극대화

### 사용자 정책 (v1.2 확정, 2026-04-30)

> "유료모델보다 로컬 LLM을 최대한 활용해서 요금을 줄이고 AI를 많이 잘 사용했으면 한다."
> "코딩은 Claude Code로 진행한다."

이 정책은 모든 후속 결정의 기준선.

**범위 정리**
- **사진 시스템 LLM**: 100% 로컬 (Ollama)
- **AI 비서·확장 활용 (§15)**: 100% 로컬 (Ollama)
- **코딩 작업**: Claude Code (구독형) — 로컬 코딩 모델은 도입하지 않음
- **유료 Vision API**: 사용 안 함 (Claude Vision/Haiku 폐기)

## 2. 하드웨어 제약 분석

### 2.1 M4 Mac mini 16GB 메모리 예산

```
통합 메모리 16GB 분배 (사진 시스템 + 트레이딩 동시 가동 시)

macOS + 시스템                    3.0~4.0 GB  (고정)
트레이딩 시스템 (유휴)             1.0~2.0 GB  (장중 +1GB)
Docker 엔진 + 컨테이너 베이스       1.5 GB
PostgreSQL (트레이딩+Immich 통합)   1.5~2.0 GB
Redis × 2                          0.3 GB
Immich Server + ML                 2.0~2.5 GB
Python 컨테이너 (OpenCV 분석)       0.5 GB
media-converter (ffmpeg 가동 시)    0.5~1.0 GB
n8n 워크플로우 실행                 0.3 GB
─────────────────────────────────────────────
고정 점유                         10.6~13.1 GB

LLM 가용 예산                      3.0~5.0 GB  ← 여기에 들어가야 함
시스템 여유 버퍼                   1.0 GB      (스왑 방지)
```

**제약 요약**: 양자화 후 모델 크기 **≤ 5GB** (보수: 4GB) 안에 모든 LLM 컴포넌트가 들어가야 함.

### 2.2 M4 GPU·Neural Engine 활용 가능성

| 가속 기술 | 활용 가능 여부 | 사용처 |
|---|---|---|
| **Metal Performance Shaders** | ✅ Ollama·llama.cpp 기본 활용 | 추론 가속 |
| **Apple Neural Engine (ANE)** | ⚠️ MLX 프레임워크만 지원 | LM Studio MLX 모드 |
| **VideoToolbox** | ✅ ffmpeg 활용 | 영상 변환 (LLM 무관) |
| **CoreML** | ✅ MLX 변환 모델 | 일부 모델 |

**중요**: M4는 ANE를 통해 추론 속도 ×1.5~2 향상 가능. **MLX framework + LM Studio** 조합이 Ollama 대비 빠름. 다만 모델 호환성은 Ollama가 더 넓음.

## 3. 사진 시스템에서 필요한 LLM 능력

### 3.1 Layer 2 (콘텐츠 분석)에서 요구되는 능력

| 능력 | 설명 | 출력 형식 |
|---|---|---|
| **장면 분류** | event/daily/scenery/food/object 5개 카테고리 | JSON |
| **객체 감지** | 케이크·꽃다발·무대·풍선·배너 (행사 오브젝트) | 객체 리스트 |
| **인물 분석** | 얼굴 수, 공식 복장 여부 (정장/드레스/한복) | bool + count |
| **음식 감지** | 음식 사진 여부 + 종류 (중복 제거 그룹핑용) | bool + tag |
| **품질 평가** | 흐림·노출·구도 보조 (OpenCV는 객관 점수만) | 1~10 점수 |
| **한국어 응답** | "행사", "음식" 등 한국어 태그 가능 | 한국어 텍스트 |

### 3.2 Layer 4 (Vision API 보조)에서 요구되는 능력

| 능력 | 설명 |
|---|---|
| **8등급 분류** | EVENT/EVENT-L/BEST/FOOD/MEMORY+/MEMORY-/NORMAL/TRASH |
| **신뢰도 출력** | 1~10 점수 (현재 Haiku 4.5 응답 형식) |
| **맥락 추론** | "친구 회식인데 정장 입고 식당" 같은 미묘한 케이스 판별 |
| **few-shot 학습** | 프롬프트 캐시 + 예시 이미지 활용 |

### 3.3 Whisper (Layer 2 음성)는 별도

음성→텍스트는 LLM과 도메인 다름. **Whisper-small 한국어 별도 유지** (이미 §17c에서 결정).

#### Llama 3.2 Vision 11B를 채택하지 않는 이유 (사용자 추천 평가)

| 항목 | 영향 |
|---|---|
| MMMU 50.7 (Qwen 58.6 대비 **-7.9**) | 분류 정확도 부족 |
| **영상 직접 입력 미지원** | Layer 2 영상 분석은 프레임별 호출 → 성능 ×N 손실 |
| 한국어 약함 | §15 한국어 자연어 생성 (앨범명·KPI) 품질 저하 |
| JSON 안정성 80% | n8n 파싱 실패 5장당 1번 |
| 11B → 6.5GB | 트레이딩 동시 가동 시 메모리 압박 |
| Llama Community License | 상업 일부 제약 |

→ **Llama 3.2 Vision 11B는 사진 시스템에 도입하지 않음**.

#### 도구 가속 (LM Studio + MLX) 분석

LM Studio + MLX = 추론 속도 ×1.5~2 (모델 성능 무관)

| 도구 | 속도 (Qwen2.5-VL 7B 1장 처리) | 운영 복잡도 |
|---|---|---|
| **Ollama (Docker)** | 5초 (Metal 가속) | 낮음 — n8n 통합 자동 |
| LM Studio (MLX) | 3초 (ANE 가속) | 높음 — 헤드리스 운영 어려움 |

- 사진 시스템 = **Ollama 권장** (Docker 통합 우선, 야간 배치 11시간 윈도에 속도 차이 미미)
- 데스크톱 자막 작업 = **LM Studio MLX** (인터랙티브 작업에 가속 체감)

> **결론**: "Llama 3.2 Vision + LM Studio MLX" 추천은 도구 부분만 일부 채택, 모델은 **Qwen2.5-VL 7B**로 교체.

## 6. 추천 (Tier 분류)

### 6.1 Tier 1 — 메인 추천: Qwen2.5-VL 7B Q4_K_M

**근거**
- 사진 시스템의 핵심은 **정확도** (사용자 사진 분류·삭제는 비가역)
- 한국어 자연 응답 (행사/음식 한국어 태그 정확)
- Apache 2.0 라이선스 (상업·재배포 자유)
- Hugging Face 다운로드 수 다수, 활발한 업데이트
- Ollama 공식 지원 (`ollama pull qwen2.5vl:7b`)

**메모리 부담 검증**
```
Qwen2.5-VL 7B Q4_K_M           4.5 GB
Whisper-small Q5_K_M           0.5 GB
─────────────────────────────────────
LLM 총합                        5.0 GB
LLM 가용 예산 (3~5 GB)          ⚠️ 상한 도달
```

**대응**: 트레이딩 장중(09:00~15:30)에는 Layer 2 분류 미실행 → 충돌 회피. 야간 16:00~05:00 윈도에서 LLM 5GB + 변환 1GB + Immich 2.5GB = 8.5GB → 16GB 중 OK.

### 6.2 Tier 2 — 안전 대안: Qwen2.5-VL 3B Q4_K_M

**근거**
- Tier 1 메모리 부담 발생 시 즉시 fallback
- 정확도는 ~10% 떨어지지만 분류 도메인에선 충분
- 추론 속도 1.5~2배 빠름

**선택 시나리오**
- Phase 0-LLM 실측 결과 메모리 압박 발견 시
- 트레이딩 시스템 메모리 사용량이 예상보다 큰 경우
- 응답 시간이 사용자 체감에 영향을 주는 경우

### 6.3 Tier 3 — 실험·미래 옵션: InternVL3 8B / Gemma 3 4B

**InternVL3 8B**
- 최신 SOTA급 vision 정확도
- 5GB로 메모리 빠듯
- 1년 후 모델 업그레이드 시 후보

**Gemma 3 4B**
- Apple Silicon MLX 최적화 우수
- LM Studio MLX 모드에서 가장 빠름
- 라이선스 (Gemma Terms) 약간 제약 있음

| **MLX (직접)** | M4 최고 성능 (×1.5~2) | 모델 변환 필요, 통합 어려움 | 🔬 실험 |

**결정**: **Ollama 메인**. 이미 사진 시스템 v3.2 §7에 포함되어 있고 n8n HTTP 노드와 호환. MLX 가속이 절실해지면 v3.3에서 LM Studio 검토.

## 8. 사진 시스템 v3.2 대체 가능성 매핑

### 8.1 Layer 2 (콘텐츠 분석) — Moondream2 → Qwen2.5-VL 7B

| 항목 | v3.2 (Moondream2 1.8B) | v3.3 후보 (Qwen2.5-VL 7B) | 개선 |
|---|---|---|---|
| 장면 분류 정확도 | ~75% | ~88% | **+13%p** |
| 한국어 응답 | 영어→번역 필요 | 한국어 직접 | UX 향상 |
| 객체 감지 (행사 오브젝트) | 6.5/10 | 8.5/10 | **+2점** |
| 음식 감지 정확도 | ~85% | ~92% | **+7%p** |
| JSON 형식 안정성 | 85% (일부 깨짐) | 95%+ | n8n 파싱 안정 |
| 사진 1장 처리 시간 | 1.5~2.5초 | 3~5초 | ❌ 2배 느림 |
| 메모리 | 1.5 GB | 4.5 GB | +3 GB |

**대체 가능 판정**: ✅ **권장**. 처리 시간 2배는 야간 배치 윈도 안에서 흡수 가능 (일 500장 × 5초 = 42분 → 야간 윈도 11시간 충분).

### 8.2 Layer 4 (Vision API 보조) — Claude Haiku → 로컬 Qwen2.5-VL 7B

가장 큰 변화. Claude API 호출 자체를 제거할 수 있는가?

**비교**

| 항목 | Claude Haiku 4.5 | 로컬 Qwen2.5-VL 7B |
|---|---|---|
| 비용 | $0.6/월 | **$0/월** |
| 8등급 분류 정확도 | ~92% (high) | ~85% (good) |
| 신뢰도 출력 | 안정적 | 프롬프트 정교화 필요 |
| few-shot 캐싱 | Anthropic 캐시 | 로컬 system 프롬프트 |
| 미묘한 맥락 추론 | 우수 | 보통 (대형 LLM 우위) |
| 응답 시간 | ~3초 (네트워크) | ~5초 (로컬 추론) |
| 외부 의존 | Anthropic API | 없음 |
| 프라이버시 | API에 사진 전송 | **로컬 처리** ✅ |

**대체 시나리오**

```
시나리오 A: 완전 대체 (Claude API 폐기)
  - 비용 절감 $0.6/월, 프라이버시 완벽
  - 정확도 -7%p → 사용자 수동수정 증가 가능
  - Phase 4 dry-run에서 검증 후 결정

시나리오 B: 하이브리드 (로컬 1차 + Claude fallback)
  - 로컬 신뢰도 < 0.5만 Claude 호출 → 월 $0.1 추정
  - 정확도 유지 + 비용 80% 절감
  - 복잡도 증가, 두 시스템 유지

시나리오 C: 로컬만 + 사용자 검토 큐
  - 로컬 신뢰도 < 0.7 자산은 Telegram 알림
  - 사용자가 5장 이내로 수동 분류
  - 비용 0, 사용자 부담 약간 추가
```

**권장 (v1.1 변경)**: **시나리오 A 우선 시도** — 로컬 LLM만으로 완전 대체

- v1.0 권장은 시나리오 C였으나, 사용자 정책(로컬 우선·비용 0)에 따라 **A 우선**
- Phase 0-LLM 실측에서 정확도 ≥ 80% 확인 후 Phase 4 dry-run 적용
- 사용자 수동수정률 추적
  - ≤ 5%: 시나리오 A 영구 유지 ⭐
  - 5~10%: 시나리오 C로 다운그레이드 (사용자 검토 큐)
  - ≥ 10%: 모델 업그레이드 (InternVL3 8B 또는 Qwen 후속) — **여전히 로컬 우선**, Claude API는 마지막 수단

**핵심**: 시나리오 B(Claude fallback)는 **선택지에서 제외**. 정확도 부족 시 더 큰 로컬 모델로 대응.

### 8.3 Layer 3 (등급 자동 판정) — 그대로 유지

규칙 기반 + n8n Function 노드 + DB 쿼리 → LLM 추론 불필요. 변경 없음.

### 8.4 Whisper (Layer 2 음성) — 그대로 유지

§17c에서 small로 결정. Qwen2.5-VL은 비전 전용, Whisper 대체 불가.

> 단, Phi-4-multimodal은 audio 입력도 지원 → 단일 모델로 통합 가능. 다만 사진 분류 정확도가 Qwen 대비 약간 낮아 v3.3에서는 별도 유지 권장.

## 15. 로컬 LLM 확장 활용 영역 (v1.1 신규)

> 사용자 정책: "로컬 LLM을 최대한 활용해서 AI를 많이 잘 사용하고 싶다"

선정된 Qwen2.5-VL 7B는 사진 분류 외에도 **텍스트 추론·자연어 생성**이 가능한 멀티모달 모델. 추가 메모리 부담 없이 다음 영역에 활용 가능 (이미 가동 중인 모델 재사용).

### 15.1 활용 영역 7가지

#### A. 앨범 이름 자동 생성 (사진 시스템 §3)

**현재 (v3.2)**: 공유 EVENT 앨범 이름은 `2026-04-29 강남구 행사` 같은 템플릿
**LLM 활용**: 앨범 내 사진 5장의 scene_tags + GPS 주소 → 자연어 앨범명 생성
**예시**: `2026-04-29 강남 결혼식` / `2026-05-01 한라산 등반` / `2026-05-15 우진이 첫 생일`
**프롬프트 예**: "이 사진들의 공통 행사를 5단어 이내 한국어 제목으로 표현"
**비용**: 일 1~2회 호출, 5초 미만

#### B. KPI 주간 리포트 자연어 요약 (Layer 7)

**현재 (v3.2)**: 표·숫자 위주 정형 텍스트
**LLM 활용**: 통계 + 변화 추세 → 자연어 인사이트 추가
**예시 출력 (LLM 추가)**:
> "이번 주 EVENT가 평소보다 30% 많았어요. 가족 행사가 잦은 한 주였네요. BEST 등급의 ♥ 추가율이 20%로 평소(12%)보다 높아 임계값 조정을 검토해도 좋겠습니다."

**비용**: 주 1회 호출, 30초

#### C. Telegram 자연어 질의 인터페이스

**현재**: 단방향 알림만
**LLM 활용**: 사용자가 자연어로 질문 → LLM이 Immich API 호출로 변환
**예시**:
- 사용자: "지난주 토요일 EVENT 사진 보여줘"
- LLM: → 날짜 추론 → Immich API → 결과 5장 첨부 응답
- 사용자: "올해 가장 잘 나온 가족 사진 5장은?"
- LLM: → BEST 등급 + face_count ≥ 2 + ♥ + 화질 점수 → 5장 응답

**구현**: n8n + Ollama + Telegram Bot Webhook
**비용**: 사용자 질의 시마다 호출, 평균 5초

#### D. 사용자 피드백 분석 (Layer 7 강화)

**현재 (v3.2)**: 등급 수동변경 패턴 → 임계값 권장값 (숫자만)
**LLM 활용**: 패턴 + 자연어 설명
**예시 출력**:
> "사용자가 '음식이 포함된 가족 모임' 사진을 자주 EVENT로 수동 승격합니다. 현재는 음식 감지 시 FOOD 분기로 가는데, face_count ≥ 3 + 음식 동시 감지 케이스는 EVENT 우선 분기를 추가하면 false negative가 줄어듭니다."

**비용**: 주 1회 호출

#### E. 사진 캡션 자동 생성 (Immich 검색 강화)

**현재**: Immich 기본 자연어 검색 (CLIP 기반)
**LLM 활용**: Layer 2 분석 시점에 한국어 캡션 동시 생성 → Immich `description` 필드 채움
**예시 캡션**: "강남구의 한 레스토랑에서 가족 4명이 케이크와 함께 찍은 사진"
**효과**:
- "케이크 사진" 자연어 검색 정확도 ↑
- 시각장애 보조용 alt-text
- 미래 통계용 텍스트 데이터

**비용**: Layer 2 호출에 추가 토큰만 사용 (별도 호출 없음, 무료)

#### F. 얼굴 그룹 이름 추천 (Immich Face Recognition)

**현재**: Immich가 얼굴 군집화는 하지만 이름은 사용자 수동
**LLM 활용**: 얼굴이 등장한 사진 5장 + 메타 → 관계 추론 → 이름 후보 제안
**예시**: "이 사람은 모든 EVENT-가족 행사에 등장하고 사용자보다 나이가 들어 보입니다. '아빠' 또는 '아버지'로 라벨링하시겠어요?"
**비용**: 신규 얼굴 군집 발견 시 1회 호출

#### G. 음식 메뉴 분류 (FOOD 등급 세분화)

**현재 (v3.2)**: FOOD는 단일 앨범
**LLM 활용**: 음식 종류별 서브 앨범 (한식/양식/일식/디저트/주류 등)
**예시 출력**: "비빔밥, 미역국 → 한식" / "스테이크, 파스타 → 양식"
**효과**: Immich에서 "최근 1년간 디저트 사진" 같은 검색 가능
**비용**: Layer 2에 추가 토큰만 (무료)
